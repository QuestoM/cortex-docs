## 1. Purpose

This Logging and Monitoring Policy defines the requirements for collecting, storing, protecting, reviewing, and retaining audit logs and monitoring data across all Questo Ltd systems and the corteX AI Agent SDK platform. Comprehensive logging and monitoring are essential for detecting security incidents, supporting forensic investigations, demonstrating regulatory compliance, and providing enterprise customers with auditable AI agent operations.

The corteX SDK includes purpose-built observability components -- AuditStream, DecisionTracer, persistent AuditLogger, and MetricsCollector -- that provide tamper-evident, hash-chained audit trails for every AI agent decision. This policy governs both Questo Ltd operational logging and the SDK's built-in audit capabilities.

**SOC 2 Mapping**: CC4.1-CC4.2 (Monitoring Activities), CC7.1-CC7.4 (System Operations)

## 2. Scope

This policy applies to:

- **Questo Ltd Operational Systems**: Cloud infrastructure, GitHub, CI/CD pipelines, communication platforms, VPN, and all other systems used in Questo Ltd business operations.
- **corteX SDK Audit Components**: The built-in audit and observability modules that provide logging capabilities to enterprise customers.
- **Personnel**: All employees, contractors, and third parties whose actions generate log data on Questo Ltd systems.
- **Log Data**: All security events, access events, change events, error events, and operational metrics generated by systems within scope.

## 3. Definitions

| Term | Definition |
|------|-----------|
| **AuditStream** | The corteX observability module (`corteX/observability/audit_stream.py`) providing tamper-evident, SHA-256 hash-chained, per-tenant audit logging |
| **DecisionTracer** | The corteX observability module (`corteX/observability/tracer.py`) recording why the agent made each decision, including alternatives considered, confidence scores, and reasoning |
| **AuditLogger** | The corteX persistent audit logger (`corteX/security/audit_logger.py`) providing file-backed, hash-chained audit entries with `AuditFileStore` |
| **DecisionLog** | The corteX engine module (`corteX/engine/decision_log.py`) recording every agent branch point with decisions, alternatives, reasoning, and outcomes |
| **MetricsCollector** | The corteX observability module (`corteX/observability/metrics.py`) collecting operational metrics (latency, token usage, costs) |
| **CostPredictor** | The corteX observability module (`corteX/observability/cost_predictor.py`) estimating and tracking LLM usage costs per tenant |
| **Hash Chain** | A sequence of audit entries where each entry includes the SHA-256 hash of the previous entry, providing tamper evidence |
| **GENESIS_HASH** | The initial hash value (`"0" * 64`) that starts each audit chain, defined in `corteX/security/audit_types.py` |
| **AuditEventType** | The enumeration of auditable event categories (LLM_CALL, TOOL_EXECUTION, POLICY_DECISION, GOAL_EVENT, SESSION_START, SESSION_END, DATA_ACCESS, CONFIG_CHANGE) |
| **SIEM** | Security Information and Event Management; a centralized log aggregation and analysis platform |
| **Log Retention** | The duration for which log data is preserved before archival or deletion |

## 4. Roles and Responsibilities

### 4.1 CTO (Policy Owner)

- Owns this policy and ensures its implementation
- Reviews monitoring alerts and escalation procedures quarterly
- Approves log retention periods and archival procedures
- Ensures SDK audit components meet enterprise customer and regulatory requirements

### 4.2 Security Lead

- Configures and maintains logging infrastructure for Questo Ltd operational systems
- Reviews security alerts and investigates anomalies on a daily basis
- Conducts monthly log review for access patterns, failed authentications, and policy violations
- Maintains log retention schedules and executes archival/deletion procedures
- Tests log integrity verification procedures quarterly

### 4.3 Engineering Team

- Ensures that corteX SDK audit components function correctly and are covered by automated tests
- Does not disable, bypass, or modify logging behavior without change management approval per POL-003
- Reports logging failures or gaps immediately to the Security Lead

### 4.4 All Personnel

- Do not attempt to tamper with, delete, or modify log data
- Report suspected logging failures or gaps to the Security Lead
- Cooperate with log-based investigations when requested

### 5.2 corteX SDK Built-In Audit Capabilities

The corteX SDK provides enterprise-grade audit capabilities through the following modules:

#### 5.2.1 AuditStream (Hash-Chained Audit Log)

`corteX/observability/audit_stream.py` provides:

- **Tamper-Evident Logging**: Each `EnterpriseAuditEntry` includes a SHA-256 hash linking it to the previous entry, forming an immutable chain from the `GENESIS_HASH`. Any modification to a past entry invalidates all subsequent hashes.
- **Per-Tenant Isolation**: Audit entries are tagged with `tenant_id`. Queries are filtered by tenant, ensuring no cross-tenant audit visibility.
- **Comprehensive Event Types**: `AuditEventType` covers LLM calls, tool executions, policy decisions, goal events, session lifecycle, data access, and configuration changes.
- **Rich Context**: Each entry captures `event_id`, `tenant_id`, `session_id`, `user_id`, `timestamp`, `action`, `justification`, `outcome`, `cost_tokens`, `data_level`, and `capabilities_used`.

#### 5.2.2 DecisionTracer (Decision Transparency)

`corteX/observability/tracer.py` provides:

- **Decision Recording**: Every model selection, tool choice, and plan step is traced with full context: alternatives considered, confidence scores, brain state, and reasoning.
- **Parent-Child Tracing**: Decisions are linked via `parent_id`, enabling tree reconstruction of the agent's decision path.
- **Performance Metrics**: Each trace includes `latency_ms` and `tokens_consumed` for cost and performance analysis.

#### 5.2.3 Persistent AuditLogger

`corteX/security/audit_logger.py` provides:

- **File-Backed Persistence**: Audit entries are written to persistent storage via `AuditFileStore` (`corteX/security/audit_storage.py`) with configurable file size limits (default 50 MB).
- **Hash Chain Integrity**: Mirrors the AuditStream hash chain, using `GENESIS_HASH` as the starting point and SHA-256 chaining for tamper evidence.
- **Severity Levels**: Entries are classified by `AuditSeverity` for filtering and alerting.
- **Callback Support**: Custom callbacks enable integration with external SIEM systems.

#### 5.2.4 AuditConfig (Tenant Logging Configuration)

`AuditConfig` (`corteX/enterprise/config.py`) provides per-tenant control over what is logged. The following configuration fields govern audit scope:

| AuditConfig Field | Default | Description | Policy Requirement |
|-------------------|---------|-------------|-------------------|
| `enabled` | `False` | Master switch for audit logging | Must be `True` for enterprise plan tenants |
| `log_messages` | `False` | Log conversation content (prompts and responses) | Defaults to off to protect confidentiality; tenant may enable for compliance |
| `log_tool_calls` | `True` | Log all tool invocations with parameters and results | Must remain `True` for SOC 2 compliance |
| `log_weight_changes` | `True` | Log synaptic weight modifications in the brain engine | Must remain `True` for processing integrity evidence |
| `log_model_routing` | `True` | Log LLM model selection decisions and routing logic | Must remain `True` for transparency and cost tracking |
| `log_destination` | `"file"` | Output target: file, syslog, or webhook | Customer-configurable; webhook enables SIEM integration |
| `log_path` | `None` | Filesystem path for file-based log storage | Required when `log_destination` is `"file"` |
| `retention_days` | `90` | Days to retain online audit logs before archival | Minimum 90 days online; 1 year in archive storage |

When `log_model_routing` is enabled, `AuditLogger._should_log()` captures all `LLM_CALL` and `LLM_FAILURE` events. When `log_tool_calls` is enabled, all `TOOL_EXECUTION` and `TOOL_FAILURE` events are captured. When `log_weight_changes` is enabled, all `WEIGHT_CHANGE` events are captured. These filters ensure that tenants can tailor audit verbosity to their compliance requirements while maintaining minimum baselines.

#### 5.2.5 DecisionLog (Engine-Level Decision Audit)

`corteX/engine/decision_log.py` provides:

- **Branch Point Recording**: Every decision the agent makes at a branch point is recorded with `DecisionType` (MODEL_SELECTION, TOOL_SELECTION, PLAN_STEP, DRIFT_RESPONSE, ESCALATION, DELEGATION, BUDGET_ADJUSTMENT, LOOP_RECOVERY, PATTERN_SELECTION).
- **Outcome Evaluation**: Decisions are rated post-hoc with `OutcomeRating` (EXCELLENT through FAILURE) enabling feedback loops.
- **Bounded Storage**: A ring buffer of 1,000 entries prevents unbounded memory growth.

### 5.3 Log Protection and Integrity

1. **Tamper Evidence**: corteX SDK audit logs use SHA-256 hash chains. Operational system logs are stored in append-only storage where available.
2. **Access Control**: Log data is classified as CONFIDENTIAL. Write access to log storage is restricted to service accounts. Human access is read-only and requires Security Lead or CTO authorization.
3. **Encryption**: Logs are encrypted at rest per POL-011. Logs in transit use TLS 1.2+.
4. **Integrity Verification**: The AuditStream hash chain is verifiable programmatically. The Security Lead runs integrity verification quarterly on persistent audit logs.
5. **Separation of Duties**: Personnel whose actions are logged cannot modify or delete those logs. Log administration is restricted to the Security Lead.

### 5.4 Log Retention

| Log Category | Retention Period | Storage Tier | Justification |
|-------------|-----------------|-------------|---------------|
| Security events (auth, access) | 12 months active + 24 months archive | Hot (12mo) / Cold (24mo) | SOC 2 evidence, incident investigation |
| Change events (code, config) | 12 months active + 24 months archive | Hot (12mo) / Cold (24mo) | Change management audit trail |
| CI/CD pipeline logs | 12 months | Hot | Build reproducibility, compliance evidence |
| VPN connection logs | 12 months | Hot | Access monitoring, incident investigation |
| corteX SDK audit logs (tenant) | 90 days online + 1 year archive (default via `AuditConfig.retention_days`); customer-configurable | Customer-managed | `AuditConfig.retention_days` default; SOC 2 minimum 90 days online |
| Operational metrics | 6 months | Hot | Performance analysis, capacity planning |

- After the retention period, logs are securely deleted using methods appropriate to the storage medium.
- Legal hold requests override retention schedules; logs subject to legal hold are preserved indefinitely until the hold is released.

### 5.5 Monitoring and Alerting

#### 5.5.1 Real-Time Alerts

The following events trigger immediate alerts to the Security Lead:

| Alert | Trigger Condition | Response SLA |
|-------|-------------------|-------------|
| Brute force attempt | 5+ failed logins in 10 minutes from same source | 1 hour |
| Unauthorized access | Access attempt to RESTRICTED resource without authorization | 30 minutes |
| Secret exposure | Secret detected in code commit or log output | 30 minutes |
| Audit chain break | Hash chain integrity verification failure | 1 hour |
| Service account anomaly | Service account used from unexpected source or at unexpected time | 2 hours |
| VPN anomaly | VPN connection from unexpected geography | 2 hours |

#### 5.5.2 Periodic Reviews

| Review | Frequency | Reviewer | Scope |
|--------|-----------|----------|-------|
| Security log review | Monthly | Security Lead | Authentication failures, access anomalies, policy violations |
| Access log review | Quarterly | Security Lead + CTO | User access patterns, privilege usage, dormant accounts |
| Change log review | Monthly | CTO | Change volume, emergency changes, rollbacks |
| Audit chain integrity | Quarterly | Security Lead | Hash chain verification on all persistent audit logs |

### 5.6 Log Aggregation

- Operational logs from all Questo Ltd systems are aggregated into a centralized logging platform.
- Log sources are normalized to a common format including: timestamp (UTC), source system, event type, user identity, action, outcome, and source IP.
- The centralized platform supports search, filtering, correlation, and alerting.
- The corteX SDK's `AuditLogger` supports callback integration, enabling customers to forward SDK audit events to their own SIEM or log aggregation systems.

## 6. Procedures

### 6.1 Daily Security Monitoring Procedure

1. The Security Lead reviews the overnight alert queue each business morning.
2. Alerts are triaged by severity: CRITICAL and HIGH alerts are investigated immediately; MEDIUM alerts are investigated within 4 hours; LOW alerts are reviewed during the monthly log review.
3. Investigation findings are documented. Confirmed security incidents are escalated per POL-004.
4. False positives are documented and alert rules are tuned to reduce noise.

### 6.2 Monthly Log Review Procedure

1. The Security Lead generates a monthly log summary report covering: total events by category, authentication failure trends, access anomalies, change activity, and alert statistics.
2. The report is reviewed by the CTO.
3. Action items (e.g., tuning alerts, investigating trends, updating access controls) are documented and tracked to completion.
4. The monthly report is retained as compliance evidence.

### 6.3 Quarterly Audit Chain Integrity Verification

1. The Security Lead runs the hash chain verification function on all persistent AuditLogger data stores.
2. Verification confirms that every entry's hash matches the computed hash of the entry content plus the previous entry's hash.
3. Results (pass/fail, entries verified, time range covered) are documented.
4. Any integrity failures trigger an immediate security incident per POL-004.

### 6.4 Log Archival and Deletion Procedure

1. At the end of each month, the Security Lead identifies log data that has exceeded its active retention period.
2. Data eligible for archival is compressed, encrypted, and moved to cold storage with a retention tag.
3. Data that has exceeded its total retention period (active + archive) is securely deleted.
4. Deletion is documented with: data category, date range covered, deletion date, and executor.
5. Legal hold exceptions are checked before any deletion.

## 7. Exceptions

1. Exceptions to this policy require written approval from the CTO.
2. Exception requests must include: the specific policy provision, business justification, compensating controls, and requested duration (maximum 6 months).
3. No exceptions are permitted for: logging of authentication events, the prohibition on log tampering, or the minimum 12-month retention period for security events.
4. All exceptions are documented in the Exception Register per POL-001 Section 7.

## 8. Enforcement

- Attempting to tamper with, delete, or modify log data is a serious violation resulting in disciplinary action up to and including termination and referral to law enforcement.
- Disabling logging on any system without change management approval per POL-003 is a policy violation.
- Failure to review logs on schedule will be identified during the quarterly security review and addressed as a process deficiency.
- The CTO reviews logging and monitoring compliance as part of the quarterly security review per POL-001.
